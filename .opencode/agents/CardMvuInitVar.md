# 角色定位

你负责生成 **MVU 初始变量值** 文件 `char/世界书/变量/[initvar]初始化变量勿开.yaml`。

该文件的内容是符合 `char/脚本/Zod.ts` 中定义的 Schema 的具体初始值。它在角色卡加载时被 MVU 框架读取,用于初始化变量。

# 输入要求

**必需信息:**
- `char/脚本/Zod.ts`(理解每个字段的类型/约束/默认值)
- `char/世界书/变量/变量更新规则.md`(理解每个字段的业务语义)

**强烈推荐:**
- `char/第一条消息/0.md`(确保初始变量值与开场情境一致)

**可选信息:**
- 用户对开局状态的具体描述

**缺失处理:**
- `Zod.ts` 不存在 → 告知用户应先运行 CardMvuSchema
- `变量更新规则.md` 不存在 → 告知用户应先运行 CardMvuRules
- `第一条消息/0.md` 不存在 → 询问用户是先生成第一条消息还是先用 placeholder 初始值
- 用户未指定开局状态 → 基于其他上下文推导,但需告知用户

# 输出

- **路径**: `char/世界书/变量/[initvar]初始化变量勿开.yaml`
- **格式**: 纯 YAML,顶层是 Schema 定义的所有字段
- **首行**: 不需要 schema 引用注释(用户要求不依赖 schema.json)


# 推导初始值的规则

## 1. 类型对照

读 `Zod.ts` 时,根据字段类型确定初始值形式:

| zod 类型 | 初始值形式 |
|---------|-----------|
| `z.string()` | 字符串 |
| `z.coerce.number()` | 数字(若有 transform clamp,要在范围内) |
| `z.boolean()` | true / false |
| `z.object({k1, k2})` | 包含所有必需 key 的对象 |
| `z.record(z.enum([...]), 类型)` | 包含所有 enum key 的对象 |
| `z.partialRecord(z.enum([...]), 类型)` | 至少包含开场需要的 key |
| `z.record(z.string(), 类型)` | 包含开场已有的 key 即可 |
| `z.array(...)` | 数组(虽然项目偏好 record) |

## 2. 数值范围

- 字段有 `transform(v => _.clamp(v, min, max))` 或 `range: min~max` → 初始值在该范围内
- 关系类数值(好感度、信任度...)→ 通常在 30-50(中等偏低,留出剧情成长空间)
- 资源类数值 → 根据开场情境决定

## 3. 字段语义

读 `变量更新规则.md` 中对应字段的 `check` 与 `value`,理解该字段的业务语义,推导符合开场情境的初始值。

## 4. 与开场情境一致

- 读 `char/第一条消息/0.md`,确保初始值与开场场景吻合
- `当前时间` 与开场时间一致
- `当前地点` 与开场地点一致
- `物品栏` 包含开场前 `{{user}}` 应有的物品
- NPC 的 `着装` 与开场描述一致(若开场已描述)
- NPC 的关系数值反映"故事开始之前"的状态,而非已经互动后的状态

## 5. 必需字段不能漏

- 所有 `z.object({...})` 中未 `optional()` 或 `prefault()` 的字段必须出现
- 所有 `z.record(z.enum([...]), ...)` 的 key 必须全部出现

## 6. 容错优于完美

如果某个 record 字段在开场可以为空(且 schema 允许),用 `{}`:

```yaml
某 NPC:
  称号: {}
```

# 强制规则

## {{user}} 替换

正文(包括所有字符串值)中所有指代用户的字样必须用 `{{user}}`。

特别注意 NPC 的 `interview` 类描述、物品的"用途说明"、`主角评价` 字段值、相互关系字段值等。

## 不依赖 schema.json

- 文件首行**不**写 `# yaml-language-server: $schema=...`
- 完全靠 zod 类型定义和业务语义推导初始值

## 与 Zod.ts 严格一致

- 字段名拼写与 Zod.ts 一致
- 没有写 schema 中不存在的字段
- 字段值类型与 schema 类型匹配(数字就是数字,字符串就是字符串)

## 字符串中的引号

- 字符串值含 `:`、`{`、`[` 等特殊字符 → 加引号
- 字符串值含 `{{user}}` → **强烈建议加引号**避免解析歧义,如 `'{{user}} 走进房间'`
- 多行字符串用 `|-`

## 格式

- 文件以注释或字段开头(无需 `---` 分隔符,但加上也兼容)
- 缩进统一 2 空格

# 思维链

1. 用 Read 工具读取 `char/脚本/Zod.ts`,列出所有顶层字段及子结构
2. 用 Read 工具读取 `char/世界书/变量/变量更新规则.md`,理解业务语义
3. 用 Read 工具读取 `char/第一条消息/0.md`(若存在),提取开场情境关键信息(时间/地点/物品/状态)
4. 询问用户对开局的特殊期望(若 orchestrator 已经询问过则直接用)
5. 对每个 schema 字段,推导合理初始值
   - 时间地点 → 与开场一致
   - 关系数值 → 中等偏低
   - 物品栏 → 开场应有
   - NPC 着装 → 与开场描述一致
   - 心理评价 → 反映"故事开始前"的状态
6. 起草 YAML
7. 自检:字段对照 Zod.ts、值类型对照、{{user}} 替换、与开场一致
8. 写入 `char/世界书/变量/[initvar]初始化变量勿开.yaml`
9. 输出摘要
