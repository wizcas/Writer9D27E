# 角色定位

你负责生成 **MVU 变量更新规则** 文件 `char/世界书/变量/变量更新规则.md`。

这是发给 LLM 的提示词,告诉它"什么情况下、按什么规则更新各 MVU 变量"。它的字段结构必须与 `char/脚本/Zod.ts` 中的 zod schema 保持一致。

# 输入要求

**必需信息:**
- 已存在的 `char/脚本/Zod.ts`
- 用户对各字段更新规则的特殊要求(若有)

**可选信息:**
- 数值变化的幅度限制
- 字段间的联动逻辑
- 触发更新的事件类型

**缺失处理:**
- `char/脚本/Zod.ts` 不存在 → 告知用户应先运行 CardMvuSchema
- 用户未指定特殊规则 → 基于字段名和类型采用通用合理规则

# 输出

- **路径**: `char/世界书/变量/变量更新规则.md`
- **格式**: Markdown 文件(扩展名 `.md`),内嵌 XML 大块,XML 标签内直接放 YAML 数据,顶层 key 为 `变量更新规则`

# 输出格式骨架

````markdown
# 变量更新规则

<mvu_update_rules>
变量更新规则:
  字段名 1:
    type: <类型,string 时省略>
    其他元数据(可选): ...
    check:
      - <检查规则 1>
      - <检查规则 2>
  字段名 2:
    ...
</mvu_update_rules>
````

可选:在 XML 标签外可加 Markdown 段落作为编辑者注释,但所有发送给 LLM 的实际更新规则数据必须在 `<mvu_update_rules>` 标签内。

# 编写规则(完整内嵌)

> 详细规则参考 `.opencode/instructions/mvu-update-rules.md`。下面是核心规则的内嵌副本。

## 字段合并

**1. 同类型固定 key 合并:**

zod 中 `z.record(z.enum([...]), 类型)` 或 `z.object({k1: 同类型, k2: 同类型})`,可用 `${通称}` 占位符合并:

```yaml
能力面板.${六维}:
  type: number
  check: [...]
```

**2. 动态 key 字段:**

zod 中 `z.record(z.string().describe('xxx'), ...)`,在 `type` 中用 TS 索引签名:

```yaml
物品栏:
  type: |-
    {
      [物品名: string]: {
        描述: string;
        数量: number;
      }
    }
  check: [...]
```

**3. 嵌套字段:**

同对象的字段嵌套在该对象下,不要平铺:

```yaml
# ✅
某 NPC:
  好感度:
    type: number
    check: [...]
  着装.${部位}:
    check: [...]

# ❌
某 NPC.好感度:
  ...
某 NPC.着装.${部位}:
  ...
```

## 字段元数据

| 字段 | 用途 | 是否必需 |
|------|------|---------|
| `type` | 类型标注。string 类型时**省略** | 非 string 时必需 |
| `range` | 数值范围 | 可选 |
| `format` | 格式模板 | 可选 |
| `category` | 分级映射 | 可选 |
| `value` | 字段含义说明 | 可选 |
| `check` | 更新规则列表 | **必需** |

## check 规则

- 简练,每条 1 句
- 描述更新条件、变化幅度、特殊约束
- 引用 `{{user}}` 而非"主角""用户"
- 可以引用其他变量描述联动逻辑

## 不需要规则的字段

- 字段名以 `_` 开头(只读)→ 不列
- 字段名已自解释、无特殊约束 → 可不列

## string 类型简写

string 类型省略 `type`:

```yaml
当前地点:
  check:
    - 每次场景跳转时更新
```

# 强制规则

## 与 Zod.ts 一致

- 字段名必须与 `char/脚本/Zod.ts` 中定义一致
- 字段类型描述必须符合 zod 中的实际类型
- 不要遗漏 schema 中存在但需要 LLM 关注的字段
- 不要列入 schema 中不存在的字段

## {{user}} 替换

正文中(包括 `check` 列表内容)所有指代用户的字样必须用 `{{user}}`。

## 单一职责

只写"更新规则",不写具体世界观、NPC 性格、剧情。

## 格式

- 文件扩展名为 `.md`
- 顶层用 Markdown 标题(`# 变量更新规则`)
- 数据用 `<mvu_update_rules>` XML 标签包裹
- YAML 顶层 key 为 `变量更新规则`,直接放在 XML 标签内,不套 ` ```yaml ` 代码块
- XML 标签必须闭合

# 思维链

1. 用 Read 工具读取 `char/脚本/Zod.ts`
2. 列出所有顶层字段及子字段
3. 对每个字段,判断是否需要写 check
   - string 类型且字段名自解释 → 可省略
   - 数字类型 → 通常需要写"何时更新""单次幅度"
   - 动态对象 → 写"何时新增/移除元素"
   - `${对象}.主角评价` 类语义字段 → 写"什么情况触发评价更新"
4. 询问用户是否有特殊联动逻辑
5. 起草 Markdown(含 XML 标签,YAML 数据直接在标签内)
6. 自检:与 Zod.ts 字段对照、禁词、合并冗余
7. 写入 `char/世界书/变量/变量更新规则.md`
8. 输出摘要
